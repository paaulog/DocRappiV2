Open Catalog

Com a Open Catalog você coloca seus produtos na Rappi de forma estruturada: primeiro descobre em qual categoria cada item se encaixa, depois registra o produto no catálogo da Rappi e, por fim, associa preço e estoque a cada loja física.

Requisitos básicos

  • Conta de parceiro e credenciais válidas. Cada chamada usa autenticação Bearer.
  • O host da API varia por país: substitua {server} pelo domínio regional (por exemplo, o mesmo usado na documentação de autenticação da Rappi para o seu mercado). Todas as URLs de catálogo começam com https://services.{server}/api/open-api/v1/catalog/.
  • Conexão HTTPS obrigatória.
  • Esta API é o ponto de partida: não depende de outras APIs de parceiro. Depois, inventário e outras integrações usam os IDs de produto e de loja que você cria aqui.

Endpoints

GET https://services.{server}/api/open-api/v1/catalog/categories

Use quando ainda não sabe quais categorias existem no topo. A resposta traz apenas o primeiro nível (a árvore completa tem três níveis). A partir daqui você combina com category-tree para achar a folha. Envie o header Authorization: Bearer {token}.

Sem corpo de requisição.

Exemplo de resposta (JSON) 
{
  "data": [
    { "id": 3569, "name": "Acessórios bebês e crianças" },
    { "id": 2900, "name": "Animais de estimação" }
  ]
}

GET https://services.{server}/api/open-api/v1/catalog/category-tree

Use para ver a hierarquia inteira de uma vez: cada nó pode ter um array children. Navegue até o último nível e anote o id da categoria folha antes de criar produto.

Sem corpo de requisição.

Exemplo de resposta (JSON) 
{
  "id": 2896,
  "name": "category-tree-br",
  "parentId": null,
  "children": [
    {
      "id": 2900,
      "name": "animais-de-estimação",
      "parentId": 2896,
      "children": []
    }
  ]
}

GET https://services.{server}/api/open-api/v1/catalog/categories/{category_id}

Chame com o id da categoria folha que você escolheu. A resposta lista attributes: use o campo code nos payloads de criação de produto e respeite isRequired e as regras de validation.

Sem corpo de requisição.

Exemplo de resposta (JSON) 
{
  "id": 1098,
  "country": "CO",
  "attributes": [
    {
      "id": 9,
      "name": "Tamaño",
      "code": "size",
      "validation": { "type": "string" }
    }
  ],
  "hasProducts": true,
  "totalProducts": 38324
}

POST https://services.{server}/api/open-api/v1/catalog/products

Cria o produto de retailer. Campos gerais: category_id (folha), name, description, has_variation, presentation (quantidade e unidade), sellType (como o item pode ser vendido). Dentro de product vão sku, ean (ou regra especial se não houver código de barras), images (URLs HTTPS públicas, PNG ou JPG, preferencialmente menores que 1 MB) e attributes alinhados aos code da categoria. Na resposta, guarde id por item e parent_id quando houver variações.

Corpo da requisição (JSON) 
{
  "category_id": 13,
  "name": "Rappi Shoes",
  "description": "Rappi Shoes",
  "has_variation": false,
  "presentation": { "quantity": 1, "unitType": "Und" },
  "sellType": {
    "type": "U",
    "minQuantity": 1,
    "maxQuantity": 10,
    "stepQuantity": 1
  },
  "product": [
    {
      "sku": "1335",
      "ean": 7702122343494,
      "name": "Rappi Shoes",
      "images": [{ "path": "https://example.com/product_0.jpg", "position": 1 }],
      "attributes": { "color_alt": "white" }
    }
  ]
}
Exemplo de resposta (JSON) 
{
  "category_id": 13,
  "name": "Rappi Shoes",
  "product": {
    "successful": [
      {
        "id": 2147484292,
        "parent_id": 151139,
        "sku": "1335",
        "ean": 730170648596
      }
    ]
  }
}

GET https://services.{server}/api/open-api/v1/catalog/products

Lista os produtos já criados para o seu retailer. Paginação por cabeçalhos limit (padrão 20, máximo 50) e offset. O retorno é resumido (útil para conferência e reconciliação); campos completos seguem o contrato vigente do endpoint na API (confira a especificação atual no seu ambiente).

Sem corpo de requisição.

Exemplo de resposta (JSON) 
{
  "data": [
    {
      "id": 2147653816,
      "sku": "31201",
      "name": "Refresco Open",
      "ean": "2200000312075",
      "status": "published",
      "country": "CO"
    }
  ],
  "limit": 50,
  "offset": 0
}

GET https://services.{server}/api/open-api/v1/catalog/products/sku?skus=SKU1,SKU2

Use quando você já tem os SKUs internos e quer buscar os registros correspondentes na Rappi em lote. Até 200 SKUs por chamada. Passe-os no query parameter skus, em geral separados por vírgula.

Sem corpo de requisição.

Exemplo de resposta (JSON) 
[
  {
    "id": 905397117,
    "sku": "13350",
    "name": "Rappi Shoes",
    "status": "published",
    "country": "BR"
  }
]

GET https://services.{server}/api/open-api/v1/catalog/stores

Primeiro passo para saber em qual “lugar” da Rappi cada unidade física sua foi cadastrada. Cada item traz id, nome, endereço e coordenadas. Paginação: limit e offset (ex.: limit=200). Cruze cada id com o código de loja no seu sistema antes de enviar estoque ou preço por loja.

Sem corpo de requisição.

Exemplo de resposta (JSON) 
{
  "data": [
    {
      "id": 8136,
      "country": "CO",
      "status": "published",
      "name": "20221213_21_Store_Name",
      "lat": "4.678318",
      "long": "-74.05496",
      "address": "Calle 93, Bogota, Colombia - ."
    }
  ]
}

Fluxos relevantes

Fluxo de criação de produto. Siga a ordem abaixo — pular etapas costuma gerar erro na API ou produto invisível no app.

Etapa 1 — Descobrir a categoria. Objetivo: saber o category_id da folha e a lista de atributos. Comece listando categorias de primeiro nível; use a árvore para descer até a folha; chame o detalhe da categoria folha para ver cada atributo (código, tipo, obrigatoriedade).

Etapa 2 — Registrar o produto. Objetivo: obter o id da Rappi para cada SKU (ou para cada variação). Envie POST em /catalog/products com a categoria, textos, imagens em URL pública HTTPS, EAN quando existir, e os atributos exigidos pela folha. Se has_variation for verdadeiro, cada variação recebe um id; todas compartilham um parent_id.

Guarde em seu banco a tabela SKU ↔ id Rappi (e parent_id se houver variações). Sem esse mapeamento, você não consegue atualizar estoque nem reconciliar pedidos.

Etapa 3 — Publicar na loja. Objetivo: o item aparecer para compra na loja certa. Liste as lojas com GET /catalog/stores, relacione cada id ao seu cadastro de unidades e envie preço e estoque por loja conforme o contrato de integração (ajustes de endereço, horário ou configuração comercial ficam com a KAM).

Depois disso, o ciclo de vida continua: alterações de preço e quantidade devem refletir na Rappi com a frequência que a operação exige, para evitar venda sem estoque ou preço desatualizado.

Guia

Visão geral

Na prática, o app da Rappi mostra ao usuário o que existe na loja mais relevante para a localização dele — não em um depósito único nacional. Por isso o catálogo tem duas dimensões: o produto “genérico” na Rappi (nome, categoria, imagens, EAN) e a oferta em cada loja (preço, disponibilidade).

A API de Catálogo é o conjunto de operações que liga seu ERP ou sistema ao modelo da Rappi. Você não escolhe categorias livremente: precisa usar a árvore de categorias fornecida pela plataforma e os atributos da categoria folha (a mais específica) para cada produto.

Resumo em três etapas: (1) descobrir a categoria correta e quais campos são obrigatórios; (2) criar o produto de retailer — a Rappi devolve um identificador que você deve guardar junto do seu SKU; (3) usar os IDs de loja da Rappi para publicar o item nas unidades certas, com preço e estoque.

Separe bem estes conceitos: SKU é o seu código interno; o id numérico devolvido pela Rappi é o que você usará em chamadas posteriores (inventário, pedidos, etc.). Um erro comum é pular a etapa de loja ou confundir categoria intermediária com categoria folha.

Conceitos-chave

Categorias funcionam como departamentos e corredores: agrupam produtos parecidos e definem quais informações o item precisa ter. Só a categoria do último nível (folha) traz a lista completa de atributos válidos para criar o produto. Categorias de níveis acima servem para navegar até essa folha.

Você obtém a lista inicial de categorias, depois a árvore completa, e por fim os detalhes de uma categoria pelo id. Não há preditor automático de categoria: a escolha da folha é sua, com base no tipo de produto.

Produto de retailer é a cópia do seu item no sistema da Rappi, com id próprio. É diferente do produto master (curado pela plataforma) e do que aparece só como “virtual” no app. Nas Open APIs, você trabalha diretamente com criação de produto de retailer e com o vínculo na loja física.

Produto na loja é o mesmo produto de retailer associado a uma loja específica, já com preço e quantidade disponível. Sem esse vínculo por loja, o usuário não vê o item como comprável naquela unidade.

Loja na Rappi corresponde a uma unidade operacional com endereço; você recebe um id de loja para cada uma e deve cruzar com o código da loja no seu sistema. Esse id entra em qualquer fluxo que seja “por loja” (estoque, disponibilidade).

Inventário (atualização contínua de preço e estoque) vem depois que o produto já existe e está ligado à loja. A API de catálogo prepara o terreno; manter dados atualizados é outro conjunto de integrações.
