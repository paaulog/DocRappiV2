Guia de integração Loyalty

Descontos diferenciados, banner de fidelidade, mapeamento no inventário (Payless / Open Orders), validação de usuário e fluxos de cadastro de novos usuários para parceiros Rappi.

Guia de integração Loyalty

O que é uma integração Loyalty?

Para oferecer uma experiência personalizada e omnicanal aos usuários Rappi que compram em um parceiro, temos programas de fidelidade como opção para que percebam todos os benefícios oferecidos pelo parceiro. Assim, um usuário frequente de um parceiro específico encontra na app Rappi os benefícios disponíveis, toma decisões de compra com base neles e aumenta o valor dos pedidos e a frequência de compra nos parceiros que disponibilizam descontos do programa de fidelidade na Rappi.

Atenção: atualmente esta integração só pode ser implementada em parceiros que já possuam uma das seguintes integrações: Connect, Payless ou Open Orders.

Checklist

  • Compreensão completa do CRM (Customer Relationship Management) que, entre outras coisas, contém dados de usuário e descontos disponíveis.
  • Fazer os ajustes necessários nos sistemas para expor as informações obrigatórias para validação.
  • Compartilhar com a Rappi os campos obrigatórios para criar novos usuários a partir da app Rappi.
Lembrete: fidelidade refere-se a programas de fidelidade e associação que oferecem descontos diferenciados. Os descontos de fidelidade que a Rappi trata hoje são apenas descontos simples (ex.: 20% de desconto). O novo esquema de integração permite automatizar todas as etapas do processo.

Componentes da integração

  • Criação de descontos diferenciados
  • Validação de usuário
  • Cadastro de novo usuário

Fluxo de fidelidade

Descontos diferenciados

Banner de fidelidade

Para gerar o banner, são necessários:

  • Uma imagem PNG com 176×132 px, até 1 MB, para o logo do programa.
  • Título do banner e corpo do banner.
  • Um endereço web (URL) dos termos e condições do programa de fidelidade do parceiro.

Mapeamento de desconto de fidelidade no PDV

Payless

No endpoint de detalhes do pedido, dentro do objeto customer, existe o campo identification. Nele aparece o ID de fidelidade do usuário. O parceiro deve extrair esse ID para aplicar corretamente os descontos de fidelidade no checkout. Se o valor for null, não deve aplicar descontos de fidelidade.

Open Orders

No evento de criação de pedido, o parceiro recebe o fidelity_id do usuário no payload do evento. Se fidelity_id estiver vazio (""), não deve aplicar descontos de fidelidade.

Descontos diferenciados no inventário

Os descontos de fidelidade podem vir na integração de inventário convencional (API ou SFTP, FULL ou DELTA).

Importante: é obrigatório ter integração de inventário já configurada e funcionando. O campo discount_price leva descontos não diferenciados. Embora a Rappi tenha salvaguardas, esses descontos devem chegar apenas uma vez no início do dia.

Se você usa integração de inventário SFTP

Envie um novo campo loyalty_discount com o desconto diferenciado do produto (percentual). O campo só é lido se loyalty_discount for maior ou igual a 0 e menor ou igual a 100. O desconto aplica-se apenas a usuários inscritos no programa de fidelidade.

Se você usa integração de inventário API

Envie um novo campo loyalty_discount com o preço diferenciado do produto. O campo só é lido se for menor que o campo price. O desconto aplica-se apenas a usuários inscritos no programa de fidelidade.

Validação de usuário

O parceiro deve validar o status de um determinado ID de fidelidade e o tipo correspondente no programa (se necessário).

O que precisamos?

  • base_url: um webhook (HTTPS) onde a Rappi possa validar as informações.
  • headers: uma chave do parceiro para que apenas a Rappi acesse o sistema do parceiro (token fixo).
Aviso: se a requisição de validação de usuário for bem-sucedida, a Rappi interpreta de imediato para exibir descontos diferenciados quando o parceiro os tiver enviado corretamente.

Endpoint de validação

GET {base_url}/loyalty/membership/{user_id}

Formato obrigatório: o endpoint deve ser implementado exatamente nesta estrutura de caminho para a integração funcionar. Valida o status de fidelidade com os dados informados na app Rappi.

Considerações

  • É necessário um header com token fixo para autenticação (você define o formato do token e o nome do header).
  • O caminho deve seguir o padrão com identificador do varejista e do membership conforme acima.
  • user_id pode ser número de ID, código de fidelidade, e-mail ou telefone — o índice que o seu sistema precisar.
  • Tempo máximo de espera: 5 segundos. Se excedido, a Rappi assume problema de conexão e cancela a requisição.
  • base_url deve usar HTTPS.
  • Este fluxo não gera access tokens; usa apenas token fixo.
A resposta deve ser 200 OK com corpo conforme um dos exemplos abaixo quando a associação for válida.
Para boa UX, a latência média deve ficar entre 200 ms e 1 s.
Requisição (curl) 
curl --location --request GET '{base_url}/loyalty/membership/{user_id}' \
--header 'X-Api-Key: {{x-api-key}}'

O user_id usado aqui é o ID armazenado do usuário e é enviado pelas outras integrações, como Open Orders e Payless.

Códigos de resposta

Código Resposta Definição
200 OK Tipo de associação (ex.: Gold, Diamond, Platinum) ou o mesmo user_id usado na validação
404 NOT FOUND ID de fidelidade não cadastrado

Corpo da resposta — 200 (níveis de associação)

JSON 
{
  "type": "membership_type"
}

Corpo da resposta — 200 (sem níveis)

JSON 
{}

Cadastro de novo usuário

Para o cadastro no programa de fidelidade, o parceiro deve compartilhar os dados necessários ao onboarding para que o usuário conclua o cadastro diretamente na app Rappi.

Aviso: se a requisição de inscrição de novo usuário for bem-sucedida, a Rappi interpreta de imediato para exibir descontos diferenciados quando o parceiro os tiver enviado corretamente.
ID vs telefone: se o parceiro exigir ID e telefone, o ID é o que aparece no endpoint de detalhes do pedido Payless.

Endpoint de inscrição

POST {base_url}/loyalty/membership

Formato obrigatório: implemente o endpoint exatamente nesta estrutura de caminho.
Latência média alvo entre 200 ms e 1 s para boa UX.

Envia os campos necessários para inscrever um novo usuário no programa de fidelidade.

Considerações

  • Token fixo em header (você define formato e nome do header).
  • Sem geração de access token; apenas token fixo.
  • Especifique quais campos precisa para o novo usuário.
  • Tempo máximo de espera: 5 segundos.
  • base_url deve usar HTTPS.
Requisição (curl) 
curl --location --request POST '{base_url}/loyalty/membership' \
--header 'X-Api-Key: {{x-api-key}}' \
--header 'Content-Type: application/json' \
--data '{
  "id": 1002485634,
  "name": "Pablo Lopez",
  "phone": "3185534870",
  "email": "pablol@gmail.com"
}'

O membership_id retornado no onboarding com sucesso é armazenado e enviado pelo Open Orders e Payless.

Respostas

Código Resposta Definição
201 CREATED ID de associação (string)
400 Bad Request Erro no corpo da requisição
406 Not Acceptable Usuário já existe

Exemplo de resposta — 201

JSON 
{
  "membership_id": "string"
}

Exemplo de resposta — 400

JSON 
{
  "type": "invalid_path_params",
  "message": "Invalid path params sent for request"
}
JSON 
{
  "type": "invalid_body",
  "message": "Invalid body sent for request"
}

Exemplo de resposta — 406

JSON 
{
  "message": "The user already exist"
}
