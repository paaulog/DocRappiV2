Inventário

Atualize estoque, preços e descontos simples na Rappi por API ou em SFTP.

Início — via API

Para atualizar inventário pela API HTTPS, há alguns pontos importantes:

O que é a Integração de Inventário?

A integração de inventário existe para facilitar a atualização do estoque e dos preços de um varejista na Rappi — neste canal, por chamadas com JSON via HTTP (datasets).

Requisitos

  • Catálogo de produtos já cadastrado na Rappi.
  • Lista de lojas com equivalência de IDs de loja entre varejista e Rappi.
  • Chave de API (api_key) para o ambiente acordado (testes ou produção).

Autenticação e formato

Envie api_key no cabeçalho. O corpo traz sobretudo o array records: cada objeto representa um produto na loja. A API pode aceitar outros campos (nome, descrição, EAN etc.), porém apenas price, discount_price e stock atualizam esses valores efetivamente do lado da Rappi conforme seu contrato.

Uma loja por requisição

Cada chamada representa o inventário de uma única loja — todos os registros devem estar alinhados a esse escopo combinado para o projeto.

🚧 Se mais de uma loja for enviada incorretamente na mesma solicitação, o endpoint poderá retornar erro.
👍 Solicite sua API key de ambiente de testes ao engenheiro de integração.

Endpoints

POST /api/cpgs-integration/datasets

Ingestão em lote de inventário (datasets). Base exemplo: https://services.grability.rappi.com/api/cpgs-integration/datasets.

Requisição (cURL) 
curl --location --request POST 'https://services.grability.rappi.com/api/cpgs-integration/datasets' \
--header 'Content-Type: application/json' \
--header 'api_key: {test_api_key}' \
--data-raw '{
    "type": "",
    "records": [
        {
            "id": "sku",
            "store_id": "store_id",
            "ean": "ean",
            "name": "productName",
            "description": "useful item",
            "trademark": "productTrademark",
            "price": 10.0,
            "discount_price": 5.0,
            "stock": 1,
            "sale_type": "U",
            "is_available": true,
            "image_url": "https://your-url.com/image"
        }
    ]
}'

📘 Coleção Postman com exemplos básicos — abrir no Postman.

Registros (campos esperados)

Tabela referência para cabeçalho e objetos dentro de records[]. Mantenha os nomes alinhados ao contrato com o seu engenheiro de integração.

Campo Exigência Tipo Escopo Definição
api_key Obrigatório string Cabeçalho Token exclusivo de integração.
type Opcional string Corpo Tipo conforme combinado (string vazia ou full/delta).
id Obrigatório string records[] SKU / PLU do parceiro.
store_id Obrigatório string records[] Identificador da loja (parceiro ou Rappi).
ean Obrigatório string records[] Código de barras (pode ser UPC).
name Obrigatório string records[] Nome do produto.
trademark Obrigatório string records[] Marca registrada.
description Obrigatório string records[] Descrição (evite vírgulas desnecessárias).
price Obrigatório número records[] Preço cheio na moeda local.
discount_price Opcional número records[] Preço final com desconto.
stock Obrigatório inteiro records[] Unidades em estoque.
sale_type Obrigatório string records[] Tipo de venda.
is_available Obrigatório booleano / null records[] Campo esperado em muitos contratos — veja observação logo abaixo.
image_url Obrigatório string records[] URL da imagem.
👍 O que vale lembrar:
  • is_available deve ser enviado quando combinado no contrato, mas tipicamente não altera a lógica de disponibilização.
  • Campos opcionais podem vir vazios ou omitidos dentro do combinado.
  • Não são aceitos campos fora da lista acordada.
  • O campo id é prioritário junto ao EAN para conciliação interna.

Carga completa (FULL)

A carga deve refletir o catálogo completo disponível na loja. Produtos ausentes ficam como indisponíveis (além dos marcados assim no payload).

Em muitos casos use corpo equivalente com "type": "" — valide os detalhes com o seu time.

Requisição (cURL) 
curl --location --request POST 'https://services.grability.rappi.com/api/cpgs-integration/datasets' \
--header 'Content-Type: application/json' \
--header 'api_key: {test_api_key}' \
--data-raw '{
    "type": "",
    "records": [
        {
            "id": "sku",
            "store_id": "store_id",
            "ean": "ean",
            "name": "productName",
            "description": "useful item",
            "trademark": "productTrademark",
            "price": 10.0,
            "discount_price": 5.0,
            "stock": 1,
            "sale_type": "U",
            "is_available": true,
            "image_url": "https://your-url.com/image"
        }
    ]
}'

Carga delta

Envie somente itens com alteração relevante desde a última sincronização. Demais permanecem como estão. Inclua "type": "delta" no corpo raiz como abaixo.

Requisição (cURL · delta) 
curl --location --request POST 'https://services.grability.rappi.com/api/cpgs-integration/datasets' \
--header 'Content-Type: application/json' \
--header 'api_key: {test_api_key}' \
--data-raw '{
    "type": "delta",
    "records": [
        {
            "id": "sku",
            "store_id": "store_id",
            "ean": "ean",
            "name": "productName",
            "description": "useful item",
            "trademark": "productTrademark",
            "price": 10.0,
            "discount_price": 5.0,
            "stock": 1,
            "sale_type": "U",
            "is_available": true,
            "image_url": "https://your-url.com/image"
        }
    ]
}'
❗️ Assim como no SFTP, não é permitido operar somente com cargas delta. Combine com ao menos uma carga completa por dia, preferencialmente no início do dia útil.

Resposta de sucesso

👍 HTTP 201 Created: o corpo traz um job_id, status e carimbo de tempo — outros canais (e-mail/notificações internas) seguem o combinado no projeto.
Exemplo de resposta (JSON) 
{
    "job_id": "83360ac9-107a-4287-a081-5c1f12f81f66",
    "status": "created",
    "created_at": "2022-11-23T16:01:58.274215827Z"
}

💡 ❗ Falta de campo obrigatório — HTTP 400:

Exemplo de resposta (JSON) 
{
    "field": "store_id",
    "description": "the field is required",
    "internal": "required",
    "row": 0,
    "row_id": null
}

💡 ❗ Token ou fonte inativa — HTTP 400 (Bucket / data source não ativo ou inexistente):

Exemplo de resposta (JSON) 
{
    "message": "data_source does not exist or not active"
}

Início — via SFTP

O que é a Integração de Inventário?

A integração de inventário existe para facilitar a atualização do estoque e dos preços de um varejista na Rappi.

Requisitos de integração

  • Catálogo de produtos já cadastrado na Rappi.
  • Lista de lojas com equivalência de IDs de loja entre varejista e Rappi.
  • Arquivo PPK (chave SSH) para autenticação SFTP.
  • Acesso ao FileZilla ou a outro cliente SFTP compatível.
  • Capacidade técnica e operacional de integrar pelo menos uma vez ao dia uma carga completa.

A solicitação (envio por arquivo)

Nesta modalidade é necessário depositar o arquivo de inventário dentro da pasta designada em nosso servidor SFTP.

👍 Múltiplas lojas em um único arquivo: é possível enviar várias lojas no mesmo arquivo.

Nome do arquivo

O nome do arquivo de integração deve seguir um dos quatro formatos abaixo. Se o envio for por loja, o nome deve incluir o ID da loja de onde o estoque está sendo reportado. Se o envio for único arquivo com todas as lojas, use o nome do parceiro.

  • Data + hora: STOREID_YYYY-MM-DD_HHmm.csv
  • Somente data: STOREID_YYYY-MM-DD.csv
  • Data + hora (todas as lojas): NOME_DO_PARCEIRO_YYYY-MM-DD_HHmm.csv
  • Somente data (todas as lojas): NOME_DO_PARCEIRO_YYYY-MM-DD.csv

Se você optar por enviar todas as lojas em um único arquivo, as linhas devem estar ordenadas por loja.

Colunas

Campo Exigência Tipo Escopo Definição
id Obrigatório string Corpo (linha) ID do produto do parceiro (SKU, PLU).
store_id Obrigatório string Corpo (linha) Código de identificação da loja do parceiro ou da Rappi.
ean Obrigatório string Corpo (linha) Código de barras (pode ser UPC).
name Obrigatório string Corpo (linha) Nome do produto.
trademark Obrigatório string Corpo (linha) Marca registrada do produto.
description Obrigatório string Corpo (linha) Descrição do produto (evite vírgulas que quebrem o CSV).
price Obrigatório número Corpo (linha) Preço cheio na moeda local.
discount_price Opcional número Corpo (linha) Preço com desconto aplicado (preço final).
stock Obrigatório inteiro Corpo (linha) Unidades reais em estoque.
sale_type Obrigatório string Corpo (linha) Tipo de venda.
image_url Obrigatório string Corpo (linha) URL da imagem.

A tabela acima mostra os campos que devem constar na integração. A seguir há um exemplo de registro mínimo, porém válido, para uma integração bem-sucedida — respeite também as convenções de nome de arquivo.

Exemplo CSV (uma linha) 
id,store_id,ean,name,trademark,description,price,discount_price,stock,sale_type,image_url
SKU-DEMO-1,900408633,7891234567890,Produto exemplo,Marca exemplo,Descrição sem vírgulas,10.99,9.49,120,U,https://cdn.exemplo.com/img.jpg

Arquivos CSV ou TXT podem chegar sem cabeçalho e sem estes nomes de coluna. Se o arquivo não trouxer cabeçalho padronizado, avise seu engenheiro de integração para mapear qual coluna corresponde a qual campo.

👍 O que vale lembrar
  • Os descontos são totalmente opcionais, mas ajudam a atrair usuários para as lojas no app Rappi.
  • Quando nem todos os produtos têm desconto, você pode repetir o preço integral em discount_price ou deixar em branco (evite enviar 0).
  • Campos não obrigatórios podem vir vazios ou podem nem ser enviados, conforme o combinado para o caso.
  • Não são aceitos campos diferentes ou ausentes da lista fornecida.
  • Embora ean seja obrigatório, nossos sistemas utilizam sobretudo o campo id para identificar produtos na base da Rappi.

Tipos de integração

Existem dois tipos de carga:

COMPLETO (FULL)

Uma atualização completa deve conter todo o catálogo disponível na loja. Qualquer produto que não apareça no arquivo será marcado como indisponível, além daqueles já indicados como indisponíveis no próprio arquivo.

Cargas completas não precisam de prefixo especial no arquivo, mas você também pode usar o prefixo convenido pela integração, por exemplo full:.

DELTA

Atualizações delta precisam trazer apenas os produtos cujo estoque (ou valores relevantes) mudou desde a última atualização. Os que não constarem no arquivo permanecem como estão no sistema.

Cargas delta devem indicar esse tipo conforme convenção técnica acordada — em geral, via prefixo delta no nome ou no envio; confirme o padrão exato com o engenheiro de integração.

❗️ Os parceiros não podem trabalhar apenas com cargas delta. Se optarem por deltas, precisam enviar ao menos uma carga completa por dia. Recomendamos fazê‑lo logo no início do dia útil.

Testando o SFTP

Seu engenheiro de integração compartilhará com você o PPK de ambiente de testes para experimentar a conexão. Em testes utilize sempre os mesmos produtos e a mesma loja combinados entre as partes.

📘 Modelo: seu engenheiro de integração disponibiliza um link de download (por exemplo «Baixar aqui») com o arquivo exemplo usado nos testes.

Após o upload, informe o engenheiro de integração para que ele verifique no pipeline se o processamento ocorreu corretamente.

❗ Importante: ao contrário da API, você não receberá resposta dentro do FileZilla apenas por concluir o upload — o resultado da integração segue pelo fluxo assíncrono notificado pela Rappi.

Notas importantes

Primeira atualização via API (produção)

Antes do go-live, confirme requisitos, testes SFTP/API e payloads com o seu engenheiro. Em produção utilize a API_KEY liberada apenas após autorização formal.

Uso combinado FULL + delta

Tanto pela API como pelo SFTP, evite trabalhar apenas com cargas delta: combine com pelo menos uma carga completa diária, de preferência pela manhã.
