Órdenes abiertas

Recibe pedidos de Rappi en tu OMS, sigue el ciclo de vida en tiempo real e integra webhooks, polling y los eventos operativos de los modos Lite o Full.

Cómo leer esta documentación
  1. Entiende el concepto, el checklist y la diferencia entre Lite y Full.
  2. Revisa el ciclo de vida y los tipos de evento (Rappi × retailer).
  3. Configura la ingesta (webhook + polling) y la gestión del pedido.
  4. Si operas en Full, sigue acuses de recibo (ACK), cambios de estado, repartidor y handshake.
  5. Si operas en Lite, usa la sección dedicada y la tabla comparativa.
  6. Consulta la referencia de cancelación cuando necesites cerrar un pedido con un motivo tipado.

Comenzando con la API de pedidos

¿Qué son las órdenes abiertas?

Open Orders permite al retailer recibir pedidos de Rappi en su propio sistema (OMS/ERP), sin depender solo de pantallas internas de la plataforma.

Mantén tu logística y operación para picking y entrega, alineado con el mismo catálogo y gobierno de stock que ya usas con Rappi.

La integración es en tiempo real: nuevos pedidos y cambios de estado viajan como eventos entre Rappi y tu backend. Los pedidos programados pueden llegar con antelación para repartir mejor la carga operativa.

¿Las Órdenes Abiertas son adecuadas para su operación?

Antes de integrar, evalúa si tu operación tiene la estructura necesaria para soportar Open Orders. Considera lo siguiente:

  • ¿Tu empresa ya cuenta con una integración funcional de inventario con Rappi?
  • ¿Es el control de inventario de la tienda lo suficientemente eficiente como para minimizar los desabastecimientos?
  • ¿Puede su sistema POS/ERP procesar pedidos de canales externos?
  • ¿Tu equipo operativo está listo para acciones urgentes y para cumplir pedidos correctamente?
  • ¿Tienes un equipo de desarrollo (interno o tercerizado) capaz de implementar, mantener y dar soporte a la integración con Rappi?
🚧 Importante: el equipo de Open Orders publica mejoras de forma continua. Como partner integrado, debes adoptar esas actualizaciones en un plazo de 8 semanas desde el lanzamiento.

Lista de verificación de integración

Antes de pasar a los puntos finales, valide estos puntos con el equipo de producto, operaciones e ingeniería:

  • Mapear elciclo vitaldel pedido y el impacto de cada evento en la app (cliente final y repartidor).
  • Traduce eventos internos de tu OMS (WMS/POS) a eventos de Rappi y viceversa.
  • Enumere las limitaciones de ERP/picking paraexcepciones(rotura, cambio, cancelación parcial) y Definir el plan B.
  • Implementa un webhook idempotente (reintentos, firma si aplica, logs) para la ingesta de pedidos.
  • Asegúrese de quecualquier cambio operativo relevanteregresar a Rappi en la versión Full (y confirmar ACK cuando sea necesario).

Modelos de operación (Lite y Full)

Open Orders ofrece dos modos. La elección define quién inicia los cambios de estado y qué acciones de API están disponibles.

Orders Lite

En Lite, Rappi concentra el picking en sus soluciones. Tu sistema recibe notificaciones y sigue el pedido, pero no modifica el flujo operativo vía API de Open Orders.

Compatible con picking de Rappi (p. ej., Shopper App, Mi Tienda, RT Picking, Cashless).

Orders Full

En Full, el retailer opera en su propio OMS y envía eventos a Rappi (picking, factura, repartidor, etc.). Hay más autonomía — y más responsabilidad de sincronización.

No es compatible con picking nativo de Rappi: la separación la conduce tu operación.

👉 ¿Dudas? Ve a LITE, FULL o la tabla comparativa más abajo.

Comprender el ciclo de vida del pedido

Descripción general del pedido

Todos los pedidos en Rappi son tratados como un flujo secuencial de eventos. Una solicitud común pasa diferentes etapas desde su creación hasta la entrega al cliente final.

Los eventos de flujo se dividen en dos tipos principales:

Eventos Rappi

Estos son eventos generados directamente por la plataforma Rappi durante el procesamiento de pedidos.

  • order_created— Notifica la creación de un nuevo pedido y envía todos información relacionada con el mismo.
  • courier_assigned— Informa que se ha asignado un repartidor (RT) al pedido y está de camino a la tienda.
  • order_finished— Indica que el pedido se entregó correctamente al cliente. fin.

Eventos para minoristas

Estos son eventos generados por el sistema de gestión de pedidos (OMS) del minorista o la solución de selección. utilizado en la operación.

  • start_picking— Notifica que el equipo de la tienda ha comenzado a separar el ordenar productos.
  • report_invoice— Informa que se ha completado la separación y que el La documentación tributaria (NF o recibo) ya está disponible según su proceso.
  • request_courier— Solicita a Rappi iniciar el proceso de atribución repartidor del pedido.

Recepción de pedidos

Rappi ofrece dos formas principales para que los minoristas reciban pedidos y actualizaciones a tiempo real:

  • Solicitar Webhook de evento
  • Punto final de sondeo

Estos métodos se pueden utilizar por separado o juntos, asegurando una mayor confiabilidad en la recepción de eventos.

Método Qué es cuando usar
Webhook Rappi envía un POST a su URL cuando haya un nuevo evento. Comocanal principal— menor latencia y mejor experiencia operativa.
Votación Tu sistema consulta periódicamente la API de Rappi. Comocontingencia(cola, fallo de red, reprocesamiento, auditoría).

Solicitar Webhook de evento

Al configurar un Webhook para pedidos de Rappi, tu operación puede recibir pedidos automáticamente tan pronto como se creen en la plataforma.

El pedido se crea a través de una solicitud.POST enviado a la URL configurada por minorista, utilizando el punto final:

Webhook (URL configurada por el socio) 
POST https://your-url.com/orders

El identificador de pedido (order_id) no se envía en la URL, sino en el cuerpo de la solicitud.

La carga útil enviada contiene toda la información relevante del pedido, que incluye:

  • Detalles de la tienda
  • Información del cliente
  • Dirección de entrega
  • Productos y combinaciones
  • Método de pago
  • Información del repartidor
  • Estado del pedido
  • Fecha de creación
  • Eventos relacionados con el pedido.

Órdenes programadas: campo place_at

el campo place_at se utiliza para identificar solicitudes programadas.

Pedidos AHORA

Cuando el pedido no esté programado, el campo se enviará vacío o nulo:

Ejemplo (JSON) 
"place_at": ""

o

Ejemplo (JSON) 
"place_at": null

Órdenes programadas

Para pedidos programados, el campo contendrá la fecha y hora programadas:

Ejemplo (JSON) 
"place_at": "2025-04-03T12:00:00Z"

Información de identificación personal

Cuando sea necesario, la información personal, como el CPF, se puede enviar en el campo.identification.number, capturado directamente en el flujo de pago de la aplicación.

Punto final de sondeo

El método de sondeo le permite consultar de forma proactiva las solicitudes y eventos activos en la plataforma Rappi.

Al realizar una solicitud GET al punto final a continuación, todas las solicitudes serán devueltas Cuestiones pendientes aún no confirmadas por el minorista:

Pedidos pendientes 
GET {country_url}/api/open-api/v1/orders

También es posible consultar un pedido concreto mediante:

Ordenar por identificación 
GET {country_url}/api/open-api/v1/orders/{order_id}

La devolución seguirá exactamente el mismo formato JSON enviado por el Webhook.

Límite de solicitud (límite de tarifa)

De forma predeterminada, los minoristas tienen un límite de:

100 solicitudes por minuto

Este límite se puede ajustar previa solicitud a Rappi.

Estado del pedido

el campo status presente tanto en el Webhook como en el punto final de sondeo indica el paso actual del solicitud dentro del flujo operativo.

Los valores posibles son:

Estado Descripción
created Orden activa, en espera de separación o en proceso de separación
invoice_reported Separación completada y canasta final de productos definida
finished Pedido entregado al cliente final.
canceled Pedido cancelado por Rappi o el minorista

¿Qué combinación utilizar en la práctica?

La elección depende de tu OMS y de la madurez de observabilidad (logs, alertas, colas). La recomendación de Rappi es clara:

  • Webhook como principal mecanismo para recibir eventos
  • Polling como mecanismo complementario y de contingencia

Si un evento no llega por webhook, el polling ayuda a detectar huecos y recuperar el estado sin depender solo de reintentos a ciegas.

Gestión de pedidos

Hay dos modelos de gestión de órdenes disponibles en Órdenes Abiertas:

FULL

En el modelo FULL, el retailer usa sus propias herramientas y sistemas (OMS) para realizar la separación de pedidos.

En este escenario, el minorista es responsable de enviar a Rappi actualizaciones sobre cada etapa del proceso operativo.

LITE

En el modelo LITE, la separación de pedidos se realiza utilizando las soluciones disponibles por la propia Rappi.

En este caso, las actualizaciones sobre eventos operativos las envía la propia plataforma al minorista.

Orden entregada

Al igual que con el proceso de asignación del conductor de entrega, la finalización del pedido también se envía a través de POST para el Webhook configurado.

Cuando se complete el pedido:

  • el campo status adquiere el valor finished
  • el campo delivered_at recibir la fecha y hora de entrega
Ejemplo (JSON) 
{
  "delivered_at": "2023-02-10T12:34:07Z"
}

Eventos en Órdenes Abiertas COMPLETAS

Esta sección aplica al modelo Full: confirmas lo que recibiste de Rappi, actualiza el estado de la solicitud en la API y completa conapretón de manosal momento de la recogida por parte del repartidor.

Orden de lectura sugerido:reconocimiento de evento (ACK) → actualización de estado (picking / NF / cancelaciones parciales) → solicitud de repartidor → apretón de manos.

Ciclo de vida del pedido: pedidos COMPLETOS

El modelo Orders FULL fue desarrollado para operaciones que utilizan su propia solución de preparación de pedidos. (recogiendo). En este escenario, el equipo de la tienda gestiona los pedidos directamente en el sistema del minorista (OMS), desde la recepción hasta la entrega al repartidor de Rappi.

Características de los pedidos COMPLETOS

Esta integración requiere que el minorista tenga:

  • Tu propio sistema de gestión de pedidos (OMS)
  • Una solución de preparación de pedidos interna
  • Equipos capacitados para ver, procesar y actualizar pedidos en tiempo real

En este modelo, el minorista es responsable de enviar a Rappi actualizaciones sobre los eventos operativos del pedido.

Confirmación del evento (acuse de recibo)

Todos los eventos enviados por Rappi deben ser confirmados por el minorista. Esto asegura que el pedido ha sido recibido y procesado correctamente.

Hay dos formas de confirmación:

Confirmación implícita

La confirmación implícita se produce a través de la respuesta del Webhook. Siempre que el minorista responda al Webhook con un código HTTP exitoso (200 a 299), el evento se considerará confirmado automáticamente.

Ejemplo de una respuesta válida:

Ejemplo (JSON) 
{
  "status": 200
}

o simplemente:

Ejemplo (JSON) 
{}

Confirmación explícita

La confirmación explícita se utiliza cuando:

  • No hay ningún Webhook configurado
  • El Webhook falla
  • La respuesta HTTP no se recibe correctamente

En este caso, debe enviar manualmente una solicitud al punto final de confirmación.

Punto final de confirmación

ACK 
POST {country_url}/api/open-api/v1/orders/acknowledge

Carga útil

Carga útil (JSON) 
{
  "acknowledgments": [
    {
      "order_id": 1234567,
      "event_uuid": "00000000-0000-0000-0000-000000000000"
    }
  ]
}
📘 Si un evento ya ha sido confirmado previamente, la API devolverá el error not_found_acknowledgements con estatus406 No Aceptable.

UUID del evento

Cada evento generado por Rappi tiene un identificador único:event_uuid.

en llamadas quealterarsolicitud, utilice siempre el UUID más reciente en el encabezado; de esta manera, Rappi se asegura de que estés reaccionando al estado actual, no a un evento antiguo.

Eventos no confirmados

Algunos eventos se consideran críticos para la operación.

Si Rappi identifica que un evento importante no fue confirmado dentro del tiempo configurado (por ejemplo, patrón,20 minutos), el pedido podrá cancelarse automáticamente.

Los principales eventos críticos son:

  • order_created
  • Eventos de modificación de productos
  • remove_product
  • remove_product_units

Actualización del estado del pedido

Al recibir el pedido, el minorista debe actualizar el progreso operativo a través de los puntos finales apropiado.

Comienzo de la separación (start_picking)

Este evento debe enviarse cuando el equipo de la tienda comience a clasificar los productos.

Punto final

Punto final 
POST {country_url}/api/open-api/v1/orders/{order_id}/start_picking

Carga útil

Carga útil (JSON) 
{
  "picking_time": 15
}

el campo picking_time representa el tiempo estimado de separación en minutos. Si no existe estimación disponible, se recomienda utilizar el valor estándar de15 minutos.

Finalización de la Separación (report_invoice)

Informa que se ha completado la separación y que elNF o reciboahora está disponible como tu proceso.

Punto final

Punto final 
POST {country_url}/api/open-api/v1/orders/{order_id}/report_invoice

Carga útil

Carga útil (JSON) 
{
  "total_price": 12.5,
  "image_url": "image.com"
}

Observaciones

  • El separador decimal debe ser "."
  • No utilices separador de miles.
  • el campo image_url es opcional
🚧 Después de enviar el report_invoice, la cesta de productos está bloqueada y ya no estará sujeta a cambios.

Cancelación de pedido

Si es necesario cancelar un pedido después de haber sido creado, utilice el siguiente punto final:

Punto final

Punto final 
POST {country_url}/api/open-api/v1/orders/{order_id}/cancel

Carga útil

Carga útil (JSON) 
{
  "type": "reason",
  "details": {}
}

Eliminación del producto (cancelación parcial)

Durante la separación, es posible que los productos no estén disponibles en stock.

Eliminación total del producto— cuando no haya unidades del producto disponibles.

Punto final 
DELETE {country_url}/api/open-api/v1/orders/{order_id}/products/{product_id}

El punto final acepta un cuerpo vacío.

Remoción Parcial de Unidades— cuando haya menos cantidad de la solicitada por el cliente.

Punto final 
PATCH {country_url}/api/open-api/v1/orders/{order_id}/products/{product_id}/quantity

Carga útil para productos unitarios

Carga útil (JSON) 
{
  "units": 2
}

Carga útil de productos por peso

Carga útil (JSON) 
{
  "grams": 250
}
🚧 Eliminar todos los productos del pedido dará lugar a la cancelación automática del pedido. Además, elimine Más del 50% de los artículos pueden requerir intervención del soporte de Rappi.

Sustituciones de productos

Asignación de repartidor

Para continuar con el flujo logístico es necesario solicitar un repartidor de Rappi.

Solicitud de persona de entrega (request_courier)

Punto final 
POST {country_url}/api/open-api/v1/orders/{order_id}/request_courier

Carga útil

Carga útil (JSON) 
{
  "store_id": 10,

  • motorbike

Ejemplo (JSON) 
"courier": {
  "phone": "18003334444",
  "last_name": "Johnson",
  "first_name": "Jane",
  "vehicle_type": "motorbike"
}

Proceso de apretón de manos

El apretón de manos es el proceso de validación que se utiliza para confirmar que el repartidor correcto está recogiendo el solicitud.

Aplica al flujo Full después de picking, facturación y asignación del repartidor: es la última barrera antes de liberar la mercancía en la tienda.

Solicitud de apretón de manos

Punto final

Punto final 
POST {country_url}/api/open-api/v1/orders/{order_id}/handshake

Respuesta

Respuesta (JSON) 
{
  "codes": [
    "033903",
    "286928",
    "788069"
  ],
  "expires_at": "2021-06-18T16:42:03Z"
}

Solo uno de los códigos es válido y será enviado al repartidor en la app de Rappi.

Reglas del apretón de manos

El apretón de manos sólo debe iniciarse cuando:

  • El pedido es completamente independiente.
  • La factura ha sido emitida.
  • Se asigna un repartidor al pedido.
  • El repartidor ya está presente en la tienda.

Además:

  • Los códigos caducan en5 minutos
  • Después del vencimiento, es necesario generar nuevos códigos.

Validación del apretón de manos

Punto final

Punto final 
POST {country_url}/api/open-api/v1/orders/{order_id}/handshake/validate

Carga útil

Carga útil (JSON) 
{
  "code": "788069"
}

Si el código es correcto:

  • La API devolverá el estado204
  • El pedido se puede entregar al repartidor.

De lo contrario, se devolverá un error.400que contiene:

  • Nuevos códigos disponibles
  • tiempo de vencimiento
  • Número restante de intentos

Códigos de error principales

Código Estado Descripción

order_not_found 404 Orden no encontrada
invalid_status 409 El estado actual no permite la operación.
handshake_already_started 208 El proceso de apretón de manos ya comenzó
no_validation_retries_left 406 Se superó el límite de intentos
no_courier_assigned 406 No hay repartidor asignado

price_difference_exceeds_percentage_limit 400 Diferencia de precio por encima del límite permitido
substitutions_not_configured 403 Anulaciones no configuradas

Eventos en Órdenes Abiertas LITE

Ciclo de vida de los pedidos: pedidos LITE

El modelo Orders LITE fue desarrollado para minoristas que desean predominantemente integración informativos o que utilizan soluciones específicas de recogida y pago.

A diferencia del modelo FULL, Órdenes LITE no permite cambios operativos en las órdenes a través del API, funcionando principalmente como mecanismo de consulta y seguimiento de eventos generados por el Rappi.

Características de Órdenes LITE

Pedidos LITE es compatible con diferentes soluciones proporcionadas por Rappi, incluyendo:

  • Aplicación de comprador
  • Mi tienda
  • Selección RT
  • Sin efectivo

En este modelo:

  • Es posible utilizar Webhooks para recibir notificaciones de pedidos.
  • Es posible consultar pedidos mediante sondeo
  • No está permitido actualizar, cancelar o modificar pedidos a través de la API.

Referencia de cancelación

Estructura básica de cancelación

Para cancelar una orden en Órdenes abiertas, utilice el siguiente punto final:

Punto final 
POST {country_url}/api/open-api/v1/orders/{order_id}/cancel

Ejemplo usando cURL:

rizo 
curl --location 'https://{COUNTRY}/api/open-api/v1/orders/{YOUR_ORDER}/cancel' \
--header 'Authorization: Bearer {YOUR_TOKEN}' \
--header 'event_uuid: {LAST_EVENT_UUID}' \
--header 'Content-Type: application/json' \
--data '{
  "type": "technical_issues",
  "details": {}
}'

Estructura de carga útil

Campo Descripción
type Motivo de la cancelación
details Información adicional relacionada con el motivo.
📘 Por la mayoría de las razones,details podría ser {}. Algunos tipos requieren información adicional.

Ejemplos de details por tipo

Orden duplicada (order_id_duplicated)

Carga útil (JSON) 
{
  "type": "order_id_duplicated",
  "details": {
    "retail_order_id": "12345",
    "created_at": "2023-01-01T00:00:00Z"
  }
}

Productos no encontrados (products_not_found)

Carga útil (JSON) 
{
  "type": "products_not_found",
  "details": {
    "products": [
      "0001",
      "0002",
      "0003"
    ]
  }
}

Productos agotados (products_stock_out)

Carga útil (JSON) 
{
  "type": "products_stock_out",
  "details": {
    "products": [
      {
        "product_id": "0001",
        "available": 0
      },
      {
        "product_id": "0002",
        "available": 1
      }
    ]
  }
}

el campo available representa la cantidad del producto aún disponible.

Diferencia de precio (products_price_difference)

Carga útil (JSON) 
{
  "type": "products_price_difference",
  "details": {
    "difference_threshold": 15,
    "products": [
      {
        "product_id": "0001",
        "price_difference": 2
      },
      {
        "product_id": "0002",
        "price_difference": 3
      }
    ]
  }
}
Campo Descripción
difference_threshold Límite porcentual permitido por diferencia de precio
price_difference Diferencia entre el precio real y el precio indicado en el pedido
