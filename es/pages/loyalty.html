Guía de integración Loyalty

Descuentos diferenciados, banner de fidelidad, mapeo en inventario (Payless / Open Orders), validación de usuario y flujos de alta de nuevos usuarios para socios Rappi.

Guía de integración Loyalty

¿Qué es una integración Loyalty?

Para ofrecer una experiencia personalizada y omnicanal a los usuarios de Rappi que compran en un socio, existen programas de fidelidad como opción para que perciban todos los beneficios del socio. Así, un usuario frecuente de un socio concreto encuentra en la app Rappi los beneficios disponibles, toma decisiones de compra basadas en ellos y aumenta el valor de los pedidos y la frecuencia de compra en socios que publican descuentos de su programa de fidelidad en Rappi.

Atención: actualmente esta integración solo puede implementarse en socios que ya tengan una de las siguientes integraciones: Connect, Payless u Open Orders.

Lista de verificación

  • Comprensión completa del CRM (Customer Relationship Management) que, entre otras cosas, contiene datos de usuario y descuentos disponibles.
  • Ajustar los sistemas para exponer la información obligatoria para la validación.
  • Compartir con Rappi los campos obligatorios para crear nuevos usuarios desde la app Rappi.
Recordatorio: «Loyalty» se refiere a programas de fidelidad y membresía con descuentos diferenciados. Los descuentos de fidelidad que Rappi maneja hoy son solo descuentos simples (p. ej. 20% de descuento). El nuevo esquema de integración permite automatizar todas las etapas del proceso.

Componentes de la integración

  • Creación de descuentos diferenciados
  • Validación de usuario
  • Registro de nuevo usuario

Flujo de fidelidad

Descuentos diferenciados

Banner de fidelidad

Para generar el banner se requiere:

  • Una imagen PNG de 176×132 px, hasta 1 MB, para el logo del programa.
  • Título del banner y cuerpo del banner.
  • Una dirección web (URL) de los términos y condiciones del programa de fidelidad del socio.

Mapeo del descuento de fidelidad en el POS

Payless

En el endpoint de detalles del pedido, dentro del objeto customer, existe el campo identification. Ahí se muestra el ID de fidelidad del usuario. El socio debe extraer este ID para aplicar correctamente los descuentos de fidelidad en el checkout. Si el valor es null, no deben aplicarse descuentos de fidelidad.

Open Orders

En el evento de creación de pedido, el socio recibe el fidelity_id del usuario en el payload del evento. Si fidelity_id está vacío (""), no deben aplicarse descuentos de fidelidad.

Descuentos diferenciados en inventario

Los descuentos de fidelidad pueden recibirse en la integración de inventario convencional (API o SFTP, FULL o DELTA).

Importante: es obligatorio tener una integración de inventario ya configurada y en funcionamiento. El campo discount_price transporta descuentos no diferenciados. Aunque Rappi tiene salvaguardas, esos descuentos deben llegar solo una vez al inicio del día.

Si usa integración de inventario SFTP

Debe enviarse un nuevo campo loyalty_discount con el descuento diferenciado del producto (porcentaje). El campo solo se lee si loyalty_discount es mayor o igual a 0 y menor o igual a 100. El descuento solo aplica a usuarios inscritos en el programa de fidelidad.

Si usa integración de inventario API

Debe enviarse un nuevo campo loyalty_discount con el precio diferenciado del producto. El campo solo se lee si es menor que el campo price. El descuento solo aplica a usuarios inscritos en el programa de fidelidad.

Validación de usuario

El socio debe validar el estado de un ID de fidelidad dado y su tipo correspondiente en el programa (si aplica).

¿Qué necesitamos?

  • base_url: un webhook (HTTPS) donde Rappi pueda validar la información.
  • headers: una clave del socio para que solo Rappi acceda al sistema del socio (token plano).
Aviso: si la solicitud de validación de usuario es exitosa, Rappi interpreta de inmediato para mostrar descuentos diferenciados cuando el socio los haya cargado correctamente.

Endpoint de validación

GET {base_url}/loyalty/membership/{user_id}

Formato obligatorio: el endpoint debe implementarse exactamente en esta estructura de ruta para que la integración funcione. Valida el estado de fidelidad con los datos de la app Rappi.

Consideraciones

  • Se requiere un header con token plano para autenticación (usted define el formato y el nombre del header).
  • La ruta debe seguir el patrón con identificador del retailer y del membership indicado arriba.
  • user_id puede ser número de ID, código de fidelidad, correo o teléfono — el índice que necesite su sistema.
  • Tiempo máximo de espera: 5 segundos. Si se supera, Rappi asume problema de conexión y cancela la solicitud.
  • base_url debe usar HTTPS.
  • Este flujo no emite access tokens; solo token plano.
La respuesta debe ser 200 OK con cuerpo conforme a uno de los ejemplos siguientes cuando la membresía sea válida.
Para buena UX, la latencia media debe estar entre 200 ms y 1 s.
Solicitud (curl) 
curl --location --request GET '{base_url}/loyalty/membership/{user_id}' \
--header 'X-Api-Key: {{x-api-key}}'

El user_id usado aquí es el ID almacenado del usuario y se envía por otras integraciones como Open Orders y Payless.

Códigos de respuesta

Código Respuesta Definición
200 OK Tipo de membresía (p. ej. Gold, Diamond, Platinum) o el mismo user_id usado en la validación
404 NOT FOUND ID de fidelidad no registrado

Cuerpo de respuesta — 200 (niveles de membresía)

JSON 
{
  "type": "membership_type"
}

Cuerpo de respuesta — 200 (sin niveles)

JSON 
{}

Registro de nuevo usuario

Para el registro en el programa de fidelidad, el socio debe compartir los datos necesarios para el alta para que el usuario lo complete directamente en la app Rappi.

Aviso: si la solicitud de inscripción de nuevo usuario es exitosa, Rappi interpreta de inmediato para mostrar descuentos diferenciados cuando el socio los haya cargado correctamente.
ID vs teléfono: si el socio requiere ID y teléfono, el ID es el que se muestra en el endpoint de detalles del pedido Payless.

Endpoint de inscripción

POST {base_url}/loyalty/membership

Formato obligatorio: implemente el endpoint exactamente en esta estructura de ruta.
Latencia media objetivo entre 200 ms y 1 s para buena UX.

Envía los campos requeridos para inscribir un nuevo usuario en el programa de fidelidad.

Consideraciones

  • Token plano en header (usted define formato y clave del header).
  • Sin generación de access token; solo token plano.
  • Especifique qué campos necesita para el nuevo usuario.
  • Tiempo máximo de espera: 5 segundos.
  • base_url debe usar HTTPS.
Solicitud (curl) 
curl --location --request POST '{base_url}/loyalty/membership' \
--header 'X-Api-Key: {{x-api-key}}' \
--header 'Content-Type: application/json' \
--data '{
  "id": 1002485634,
  "name": "Pablo Lopez",
  "phone": "3185534870",
  "email": "pablol@gmail.com"
}'

El membership_id devuelto cuando el alta tiene éxito se almacena y se envía por Open Orders y Payless.

Respuestas

Código Respuesta Definición
201 CREATED ID de membresía (string)
400 Bad Request Error en el cuerpo de la solicitud
406 Not Acceptable El usuario ya existe

Ejemplo de respuesta — 201

JSON 
{
  "membership_id": "string"
}

Ejemplo de respuesta — 400

JSON 
{
  "type": "invalid_path_params",
  "message": "Invalid path params sent for request"
}
JSON 
{
  "type": "invalid_body",
  "message": "Invalid body sent for request"
}

Ejemplo de respuesta — 406

JSON 
{
  "message": "The user already exist"
}
