Inventario

Actualiza stock, precios y descuentos simples en Rappi vía API o SFTP.

Inicio — vía API

Para actualizar inventario por API HTTPS, hay algunos puntos importantes:

¿Qué es la Integración de Inventario?

La integración de inventario existe para facilitar la actualización de stock y precios de un minorista en Rappi — en este canal, mediante llamadas JSON por HTTP (datasets).

Requisitos

  • Catálogo de productos ya registrado en Rappi.
  • Lista de tiendas con equivalencia de IDs entre el minorista y Rappi.
  • Clave de API (api_key) para el entorno acordado (pruebas o producción).

Autenticación y formato

Envía api_key en el header. El cuerpo contiene principalmente el arreglo records: cada objeto representa un producto de la tienda. La API puede aceptar otros campos (nombre, descripción, EAN, etc.), pero solo price, discount_price y stock actualizan efectivamente esos valores del lado de Rappi, según tu contrato.

Una tienda por solicitud

Cada llamada representa el inventario de una sola tienda — todos los registros deben estar alineados con ese alcance acordado para el proyecto.

🚧 Si se envía más de una tienda incorrectamente en la misma solicitud, el endpoint puede devolver error.
👍 Solicita tu API key del entorno de pruebas al ingeniero de integración.

Endpoints

POST /api/cpgs-integration/datasets

Ingesta masiva de inventario (datasets). URL base de ejemplo: https://services.grability.rappi.com/api/cpgs-integration/datasets.

Solicitud (cURL) 
curl --location --request POST 'https://services.grability.rappi.com/api/cpgs-integration/datasets' \
--header 'Content-Type: application/json' \
--header 'api_key: {test_api_key}' \
--data-raw '{
    "type": "",
    "records": [
        {
            "id": "sku",
            "store_id": "store_id",
            "ean": "ean",
            "name": "productName",
            "description": "useful item",
            "trademark": "productTrademark",
            "price": 10.0,
            "discount_price": 5.0,
            "stock": 1,
            "sale_type": "U",
            "is_available": true,
            "image_url": "https://your-url.com/image"
        }
    ]
}'

Registros (campos esperados)

Tabla de referencia para el encabezado y objetos dentro de records[]. Mantén los nombres alineados con el contrato acordado con tu ingeniero de integración.

Campo Requisito Tipo Alcance Definición
api_key Obligatorio string Header Token exclusivo de integración.
type Opcional string Body Tipo según lo acordado (string vacía o full/delta).
id Obligatorio string records[] SKU / PLU del partner.
store_id Obligatorio string records[] Identificador de la tienda (partner o Rappi).
ean Obligatorio string records[] Código de barras (puede ser UPC).
name Obligatorio string records[] Nombre del producto.
trademark Obligatorio string records[] Marca registrada.
description Obligatorio string records[] Descripción (evita comas innecesarias).
price Obligatorio número records[] Precio completo en moneda local.
discount_price Opcional número records[] Precio final con descuento.
stock Obligatorio entero records[] Unidades en stock.
sale_type Obligatorio string records[] Tipo de venta.
is_available Obligatorio booleano / null records[] Campo esperado en muchos contratos — ver nota abajo.
image_url Obligatorio string records[] URL de la imagen.
👍 Qué tener en cuenta:
  • is_available debe enviarse cuando se acuerde en el contrato, pero normalmente no altera la lógica de disponibilidad.
  • Los campos opcionales pueden venir vacíos u omitidos según lo acordado.
  • No se aceptan campos fuera de la lista acordada.
  • El campo id es prioritario junto con el EAN para conciliación interna.

Carga completa (FULL)

La carga debe reflejar el catálogo completo disponible en la tienda. Los productos ausentes quedan como no disponibles (además de los marcados así en el payload).

En muchos casos usa un cuerpo equivalente con "type": "" — valida los detalles con tu equipo.

curl --location --request POST 'https://services.grability.rappi.com/api/cpgs-integration/datasets' \
--header 'Content-Type: application/json' \
--header 'api_key: {test_api_key}' \
--data-raw '{
    "type": "",

            "id": "sku",
            "store_id": "store_id",

Carga delta

Envía solo ítems con cambios relevantes desde la última sincronización. Los demás permanecen igual. Incluye "type": "delta" en el cuerpo raíz como abajo.

Solicitud (cURL · delta) 
curl --location --request POST 'https://services.grability.rappi.com/api/cpgs-integration/datasets' \
--header 'Content-Type: application/json' \
--header 'api_key: {test_api_key}' \
--data-raw '{
    "type": "delta",
    "records": [
        {
            "id": "sku",
            "store_id": "store_id",
            "ean": "ean",
            "name": "productName",
            "description": "useful item",
            "trademark": "productTrademark",
            "price": 10.0,
            "discount_price": 5.0,
            "stock": 1,
            "sale_type": "U",
            "is_available": true,
            "image_url": "https://your-url.com/image"
        }
    ]
}'
❗️ Igual que en SFTP, no está permitido operar solo con cargas delta. Combínalo con al menos una carga completa por día, preferentemente al inicio de la jornada.

Ejemplo de respuesta (JSON)

    "message": "data_source does not exist or not active"
}

Inicio — vía SFTP

¿Qué es la Integración de Inventario?

La integración de inventario existe para facilitar la actualización del stock y precios de un minorista en Rappi.

Requisitos de integración

  • Catálogo de productos ya registrado en Rappi.
  • Lista de tiendas con equivalencia de IDs entre el minorista y Rappi.
  • Archivo PPK (clave SSH) para autenticación SFTP.
  • Acceso a FileZilla u otro cliente SFTP compatible.
  • Capacidad técnica y operativa para integrar al menos una carga completa al día.

La solicitud (envío por archivo)

En esta modalidad es necesario depositar el archivo de inventario en la carpeta designada de nuestro servidor SFTP.

👍 Múltiples tiendas en un único archivo: es posible enviar varias tiendas en el mismo archivo.

Nombre del archivo

El nombre del archivo de integración debe seguir uno de los cuatro formatos siguientes. Si el envío es por tienda, el nombre debe incluir el ID de la tienda desde donde se reporta el stock. Si el envío es un único archivo con todas las tiendas, usa el nombre del partner.

  • Fecha + hora: STOREID_YYYY-MM-DD_HHmm.csv
  • Solo fecha: STOREID_YYYY-MM-DD.csv
  • Fecha + hora (todas las tiendas): NOMBRE_DEL_PARTNER_YYYY-MM-DD_HHmm.csv
  • Solo fecha (todas las tiendas): NOMBRE_DEL_PARTNER_YYYY-MM-DD.csv

Si eliges enviar todas las tiendas en un único archivo, las líneas deben estar ordenadas por tienda.

Columnas

Campo Requisito Tipo Alcance Definición
id Obligatorio string Body (fila) ID de producto del partner (SKU, PLU).
store_id Obligatorio string Body (fila) Código de identificación de tienda del partner o de Rappi.
ean Obligatorio string Body (fila) Código de barras (puede ser UPC).
name Obligatorio string Body (fila) Nombre del producto.
trademark Obligatorio string Body (fila) Marca registrada del producto.
description Obligatorio string Body (fila) Descripción del producto (evita comas que rompan el CSV).
price Obligatorio número Body (fila) Precio completo en moneda local.
discount_price Opcional número Body (fila) Precio con descuento aplicado (precio final).
stock Obligatorio entero Body (fila) Unidades reales en stock.
sale_type Obligatorio string Body (fila) Tipo de venta.
image_url Obligatorio string Body (fila) URL de la imagen.

La tabla anterior muestra los campos que deben incluirse en la integración. A continuación hay un ejemplo mínimo, pero válido, para una integración exitosa — respeta también las convenciones del nombre de archivo.

Ejemplo CSV (una fila) 
id,store_id,ean,name,trademark,description,price,discount_price,stock,sale_type,image_url
SKU-DEMO-1,900408633,7891234567890,Producto ejemplo,Marca ejemplo,Descripción sin comas,10.99,9.49,120,U,https://cdn.ejemplo.com/img.jpg

Los archivos CSV o TXT pueden llegar sin encabezado y sin estos nombres de columna. Si el archivo no trae encabezado estandarizado, avisa a tu ingeniero de integración para mapear qué columna corresponde a qué campo.

👍 Qué tener en cuenta
  • Los descuentos son totalmente opcionales, pero ayudan a atraer usuarios a las tiendas en la app de Rappi.
  • Cuando no todos los productos tienen descuento, puedes repetir el precio completo en discount_price o dejarlo en blanco (evita enviar 0).
  • Los campos no obligatorios pueden venir vacíos o incluso no enviarse, según lo acordado para el caso.
  • No se aceptan campos diferentes o faltantes respecto a la lista provista.
  • Aunque ean es obligatorio, nuestros sistemas usan sobre todo el campo id para identificar productos en la base de Rappi.

Tipos de integración

Existen dos tipos de carga:

COMPLETO (FULL)

Una actualización completa debe contener todo el catálogo disponible en la tienda. Cualquier producto que no aparezca en el archivo se marcará como no disponible, además de los ya marcados como no disponibles en el propio archivo.

Las cargas completas no necesitan prefijo especial en el archivo, pero también puedes usar el prefijo acordado para la integración, por ejemplo full:.

DELTA

Las actualizaciones delta deben incluir solo los productos cuyo stock (o valores relevantes) cambió desde la última actualización. Los que no aparezcan en el archivo permanecen igual en el sistema.

Las cargas delta deben indicar ese tipo según la convención técnica acordada — en general, mediante prefijo delta en el nombre o en el envío; confirma el patrón exacto con el ingeniero de integración.

❗️ Los partners no pueden trabajar solo con cargas delta. Si optan por delta, deben enviar al menos una carga completa por día. Recomendamos hacerlo al inicio de la jornada.

Pruebas de SFTP

Tu ingeniero de integración compartirá contigo el PPK del entorno de pruebas para probar la conexión. En pruebas, utiliza siempre los mismos productos y la misma tienda acordados entre las partes.

📘 Plantilla: tu ingeniero de integración te comparte un enlace de descarga (por ejemplo "Descargar aquí") con el archivo de ejemplo usado en pruebas.

Después del upload, informa al ingeniero de integración para que verifique en el pipeline si el procesamiento ocurrió correctamente.

❗ Importante: a diferencia de la API, no recibirás respuesta dentro de FileZilla solo por finalizar el upload — el resultado de la integración sigue el flujo asíncrono notificado por Rappi.

Notas importantes

Primera actualización vía API (producción)

Antes del go-live, confirma requisitos, pruebas SFTP/API y payloads con tu ingeniero. En producción usa la API_KEY liberada solo tras autorización formal.

Uso combinado FULL + delta

Tanto por API como por SFTP, evita trabajar solo con cargas delta: combínalas con al menos una carga completa diaria, preferentemente por la mañana.
