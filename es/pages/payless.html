Rappi Payless — Guía de integración

Documentación técnica de Payless para sistemas punto de venta (POS) e integradores: flujos operativos, APIs de sandbox y producción, autenticación, autorización, conciliación y buenas prácticas de integración con Rappi.

Guía de integración Rappi Payless

Cómo navegar
  1. Lea la política, qué es Payless y las consideraciones técnicas.
  2. Revise el flujo, el PDF y la lista de verificación de integración.
  3. Siga la trayectoria del pedido (pasos 1–9) y las reglas del código de barras.
  4. Implemente sandbox, JWT, detalles del pedido y authorize.
  5. Si aplica, implemente Price Match; use las tablas de errores, cancelación, reembolso y conciliación.

Política y restricciones

Importante: para garantizar una excelente experiencia a usuarios finales y a RTs/personal shoppers, los socios no pueden implementar validaciones internas que impidan el envío de la solicitud de autorización ni desarrollos no descritos en esta documentación. Si ocurre ese caso, a pesar de esta recomendación, las tiendas no migrarán a Payless hasta que el socio elimine esas validaciones y desarrollos extra del código.

¿Qué es Rappi Payless?

Rappi Payless es un método de pago sin tarjeta y sin efectivo en tienda, exclusivo para nuestros pickers (personal shoppers y repartidores RT).

Permite que los sistemas punto de venta (POS) procesen pedidos Rappi con códigos de pago generados en las apps móviles de los pickers. Esos códigos se muestran como código de barras o QR.

Payless elimina comisiones por transacción con tarjeta, simplifica la conciliación y los informes, reduce el tiempo de checkout y evita problemas operativos ligados a la liberación de fondos en tarjetas de débito.

Consideraciones técnicas de Payless

  • Payless expone una API pública con raíz por país en Latinoamérica; la misma integración puede usarse en todos los países.
  • Funciona en HTTPS TLS 1.2+ con tokens JWT. Por reglas de proceso, Payless puede continuar flujos de pago aun con interrupciones de comunicación.
  • El código de aprobación puede ser QR, código de barras o numérico; las tres opciones están siempre disponibles.
  • Es posible consultar a diario información de ventas para cruzarla con informes generados en el POS mediante una API.

Lista de verificación de integración

Tras implementar con éxito Rappi Payless, se espera que su solución:

  • Ingeste y procese códigos Payless en código de barras, QR o entrada manual.
  • Envíe solicitudes de autorización de pago a Rappi con todos los detalles obligatorios.
  • Maneje correctamente transacciones aprobadas y rechazadas.
  • Maneje timeouts con reintentos automáticos.
  • Cuando el pago no sea posible vía Payless (red móvil, fallas de internet, etc.), permita pagar el pedido con tarjeta o efectivo.
  • Disponer de un listado diario de las transacciones autorizadas por Rappi.

Introducción

Esta guía presenta una visión general de Rappi Payless. Encontrará información sobre:

  • La experiencia del picker con Payless
  • El flujo normal de pago y el flujo alternativo (fallback)
  • Cómo obtener datos para alimentar el proceso de conciliación
  • Cómo usar los métodos de sandbox para configurar pruebas manuales y automatizadas

El cliente hace un nuevo pedido — pasos operativos

  1. El cliente hace un nuevo pedido. Según la configuración de la tienda, un algoritmo decide si el pedido lo cumple un personal shopper o un repartidor (RT).
  2. Picking de productos. En la lista de productos, el picker debe marcar cada artículo como «encontrado», «no encontrado» o «sustituido». No puede avanzar al pago con productos pendientes en la lista.
  3. Elección del método de pago. En esta fase la app del picker está en modo pago, generando nuevos códigos cada 10 minutos.
  4. (Si existe flujo Price Match) El POS obtiene el carrito Rappi y los datos del cliente. Es necesario obtener los detalles del pedido, comparar precios Rappi vs POS y, si hace falta, sobrescribir los que difieran. También sirve cuando el POS necesita identificación del usuario para puntos de fidelidad. El código de pago se lee en el dispositivo del picker para obtener los detalles del pedido.
  5. La caja registra todos los productos. El POS debe registrar el documento del usuario para acumulación de puntos. Mientras tanto la caja registra todos los productos como en el flujo habitual.
  6. (Si existe Price Match) El POS sobrescribe los precios. El POS debe alinear los precios de los artículos registrados con los obtenidos previamente del pedido Rappi, cuando sean diferentes.
  7. Lectura del código de pago en el dispositivo del picker. La caja puede leer QR, código de barras o introducir el código manualmente. Si el POS ya leyó el código al obtener la identificación del cliente antes de registrar los productos, debe volver a leer el código: tiene tiempo de vida y puede haberse renovado.
  8. El POS envía los detalles de la compra a Rappi. Se trata de una solicitud HTTP al endpoint de autorización de pago. En caso de éxito, el POS recibe un token de autorización de transacción en la respuesta.
  9. El picker fotografía el ticket. El pedido queda listo para entrega.

El código de barras Payless de Rappi

El código de barras Payless (también llamado código de pago) es un número aleatorio asociado a un identificador único de pedido en nuestra base de datos. El formato lineal es CODE-128.

El tiempo de vida (TTL) total del código de barras es de 10 minutos; en las apps de los pickers el código se renueva cada 10 minutos en la vista de pago.

Reintentos automáticos en timeouts: el TTL es de 10 minutos. Téngalo en cuenta al implementar reintentos automáticos por fallas de comunicación o timeouts entre el POS y el backend de Rappi.

Usos del código de barras

Además de la solicitud de autorización de pago, el código sirve para obtener detalles del pedido, cancelaciones y reembolsos.

Proceso Payless — autorización de pago

Una transacción exitosa siempre devuelve HTTP 200 e incluye un token de autorización en el cuerpo de la respuesta. El código de barras y el token (o código de autorización) de cada transacción exitosa deben guardarse del lado del retailer para conciliación.

Autorizaciones rechazadas

Hay pocas causas de rechazo, aún sin códigos de motivo estables, y no se espera que el POS exponga esa información al cliente. En cada rechazo, el picker recibe notificación push con instrucciones para ajustar el pedido antes de volver a intentar.

Solicitud de autorización mal formada

Pueden faltar propiedades obligatorias o existir tipos inconsistentes en la solicitud de autorización.

El monto solicitado supera el umbral autorizado

Puede deberse a diferencias de precio no comunicadas que elevan el monto por encima del umbral, o a productos pedidos vía chat que el picker no incluyó correctamente. En cada rechazo, los pickers reciben push indicando inconsistencias; tras el ajuste, el pedido no debería volver a rechazarse por el mismo motivo.

Pedidos inaceptables

Pedidos con error unacceptable_order pueden haber sido cancelados por el cliente (app del picker sin sincronizar), ya estar pagados, o el repartidor intentar pagar en otro socio retailer.

Códigos inválidos

A menudo asociados a intentos de fraude. Se captura pantalla del código; el pago se rechaza por código expirado o ya aplicado a otro pago.

Tienda incorrecta

Si el repartidor intenta pagar en una tienda distinta de la pedida en la app Rappi, la autorización falla hasta ir a la tienda correcta o hasta que el pedido se libere para otro repartidor.

Productos inválidos

Si el repartidor intenta pagar productos distintos del pedido original, no se autorizará hasta entregar en caja los productos correctos del pedido.

Fallas de comunicación

Buenas prácticas Payless: implementar reintento automático en el POS cuando no haya respuesta a la solicitud de autorización de pago.

Ejemplo: la caja dispara la solicitud de autorización en el POS. Se recomienda timeout de unos 20 segundos y hasta 5 intentos automáticos dentro del TTL del código de barras. El timeout es configurable según la latencia de red del retailer.

Buenas prácticas — riesgo de doble autorización

La solicitud puede llegar a los servicios de Rappi y la autorización ser exitosa, pero la respuesta no llegar al POS: la app RT muestra pagado y la caja no imprime. El socio debe implementar cancelación de la autorización en el endpoint adecuado para evitar doble pago (débito/efectivo y Payless), pues el RT puede acabar pagando por otro medio.

Alternativa: tarjeta de débito / efectivo

Debe ser una excepción rara. El flujo alternativo con tarjeta es obligatorio, pero debe ser raro. El KPI principal de Payless es procesar ~99,9% de las transacciones en el motor de autorización — el fallback debe ocurrir cerca de 1 en 1000 pedidos.

El sistema de fallback se rediseñó con reglas y reintentos: tras fallas, la app RT/Shopper cambia automáticamente a pago con tarjeta o efectivo, sin intervención del socio en medio del pago.

Nota: la versión con PIN está discontinuada.

Autenticación (DEV — PROD)

Las APIs CPG de Rappi usan autenticación JWT. Intercambie las credenciales proporcionadas por Rappi por un token de acceso para autenticar las solicitudes. El token expira a las 12 horas; cuando expire, solicite uno nuevo en el endpoint de sign-in.

  • Todas las solicitudes a la API deben usar HTTPS. Las llamadas en HTTP plano fallan.
  • Las solicitudes sin token de acceso válido fallan.
Nota: las credenciales de integración son sensibles — manténgalas seguras. No las comparta en repositorios públicos, código de cliente, chats, etc. En caso de sospecha de filtración, contacte a Rappi de inmediato para revocar secretos y emitir nuevas credenciales. Un abuso sospechoso de la API puede llevar a la suspensión del acceso hasta aclararlo.

Consideraciones

  • El token solo es válido 12 horas.
  • Una solicitud puede cambiar de token durante el flujo Payless.
  • El token creado en su ambiente solo es válido en ese ambiente.

POST /api/cpgops-integrations/retailers/sign-in

Genera tokens JWT de acceso. Devuelve un token para autenticar sus solicitudes durante las próximas 12 horas.

Obtener detalles del pedido (DEV — PROD)

En cualquier pedido, todos los campos del cuerpo de respuesta pueden llegar a null. Este endpoint no puede usarse después de la autorización de pago.

Debe llamarse antes de la solicitud de autorización. Incluye carrito (SKU, precio, unidades, etc.), order_id, store_id y cliente cuando el POS lo necesite.

Fidelidad / descuentos simples: Rappi envía en customer.identification el identificador de fidelidad del usuario validado cuando no es nulo — cárguelo en el POS para aplicar descuentos de fidelidad correctamente.

GET /api/cpgops-integrations/pos/v3/payments/{barcode}/order

Solicitud (curl)
curl --location 'https://microservices.dev.rappi.com/api/cpgops-integrations/pos/v3/payments/{barcode}/order' \
--header 'Authorization: {access-token}'

Los encabezados pueden incluir AUTH_USER / user según el contrato de integración.

Forma típica de la respuesta: order_id, currency, amount, store_id, customer, delivery_address, basket[] con precios unitarios y totales.

Información de factura del usuario (solo PROD)

Los campos de factura pueden ser nulos — Rappi no obliga al usuario a completar estos datos. Si son nulos o vacíos, emita factura a «consumidor final» / «anónimo». Los datos de factura electrónica pueden venir en customer.electroninc_invoice_* (ortografía según la API).

Autorización de pagos con Payless (DEV - PROD)

Solicita la autorización de pago con el código de barras del RT/picker. En caso de éxito, un código de transacción confirma la autorización. El store_id debe coincidir con la integración de inventario Rappi. terminal_id y terminal_transaction_id son identificadores únicos del socio.

El carrito (basket) es obligatorio: todos los productos registrados en caja. Metadata es libre. El amount es el total final tras descuentos, alineado con la integración de inventario.

¿Qué es el basket?

  • sku debe coincidir con el inventario Rappi.
  • quantity: número de unidades o peso en KG para pesables.
  • unit_type: igual al inventario — U (unidad), WW (pesable suelto), WB (pesable empaquetado).
  • taxes: descripción por línea (p. ej. IVA 19%) para conciliación financiera.
  • Agrupe líneas idénticas; los impuestos deben reflejar el agrupamiento.

[VERSIÓN PRICE MATCH] Autorización de pagos con Payless (DEV - PROD)

Desarrolle este flujo solo si el ingeniero de integración lo indica. Si le indicaron el flujo legado, siga la documentación legada proporcionada por Rappi.

Es el mismo endpoint de autorización; aplican reglas extra de carrito y precios alineados a los detalles del pedido Rappi.

Price Match

El socio debe obtener los precios de la app Rappi (detalles del pedido), sobrescribirlos en el POS durante el pago e imprimir el documento fiscal con los precios alineados.

Consideraciones

  • Con Price Match en alcance, no procese ventas Payless sin esta capacidad.
  • No siempre es posible alinear (p. ej. pesables con empaque aproximado) — puede mantener el precio de tienda.
  • Ítems solo en el POS y fuera del pedido Rappi: regístrelos con precio de tienda e inclúyalos en el carrito de autorización.
  • Ítems solo en el pedido Rappi y no registrados: no procese; no los incluya en el carrito de autorización.

Lógica de Price Match

  • Si SKU (o EAN) no coincide: use precio y precio unitario del POS.
  • Si coincide — producto unitario: use precio y precio unitario Rappi.
  • Si coincide — pesable: use precio por kg Rappi; línea = precio (kg) Rappi × cantidad registrada en el POS.
Nota: precio calculado = precio Rappi (kg) × cantidad en el POS (vea la figura de pesable en la trayectoria).

Ejemplos (resumen)

  1. Caso 1: dos productos, ambos hacen match, Rappi más caro — use precios Rappi en ambos.
  2. Caso 2: una línea no hace match — solo SKUs coincidentes usan precio Rappi.
  3. Caso 3: una línea Rappi, dos en el POS — Rappi gana donde hay match; resto POS.
  4. Caso 4: conteos de unidades distintos (riesgo chat/fraude) — use unidades POS y ajuste el total; puede bloquear hasta que el picker corrija.
  5. Caso 5: diferencias en pesables — final = precio kg Rappi × peso en el POS.

Use la misma forma de solicitud POST .../authorize del flujo estándar.

Endpoint de validación de pago

Cuando falla la comunicación y el POS no sabe si la autorización tuvo éxito, valide el estado del pago del pedido.

GET /api/cpgops-integrations/pos/v3/payments/orders/{order_id}/status

Debe usarse después de la solicitud de autorización (error si se usa antes).

Solicitud (curl)
curl --location 'https://microservices.dev.rappi.com/api/cpgops-integrations/pos/v3/payments/orders/{order_id}/status' \
--header 'Authorization: {access-token}'

La respuesta incluye estado del pago, carrito, código de autorización, identificadores de tienda y terminal, etc.

Cancelación de transacciones (DEV - PROD)

Importante: tras cancelar, mantenga flujo para volver a pagar con Payless u otro método. La cancelación revierte la autorización, no el pedido Rappi.

Úselo cuando Rappi autorizó Payless pero la respuesta de éxito no llegó al POS (riesgo de doble pago con dispersión/tarjeta). No imprima ni entregue productos hasta que el estado quede claro. Se recomienda al menos 5 reintentos tras 20 segundos sin respuesta antes de decidir.

Guarde order_id y código de barras tras autorización exitosa. El JSON debe coincidir con la solicitud de autorización. Solo el mismo día (hasta 23:59:59 GMT 0).

POST /api/cpgops-integrations/pos/v3/payments/{barcode}/cancel

Misma estructura de cuerpo que authorize (más order_id si el contrato lo exige — confirme con Rappi).

Solicitud (curl)
curl --location 'https://microservices.dev.rappi.com/api/cpgops-integrations/pos/v3/payments/{barcode}/cancel' \
--header 'Content-Type: application/json' \
--header 'X-Rappi-Channel: B2B' \
--header 'Authorization: {access-token}' \
--data '{ ... mismos campos que authorize ... }'

En éxito, el estado puede evolucionar por tipos de transacción de cancelación descritos en las especificaciones de Rappi.

Reembolsos (DEV - PROD)

El pedido debe estar previamente autorizado; siga las reglas de cancelación en el recorrido operativo. En este momento solo se admiten reembolsos totales.
  • Use los mismos store_id, terminal_id, terminal_transaction_id de la autorización.
  • El cuerpo debe coincidir con la autorización; el reembolso total usa carrito vacío [] según la especificación.
  • El mismo día de la autorización; mismo código de barras de la autorización exitosa.
  • Reembolso total cuando el pedido se cancela y el RT devuelve productos a la tienda — no para devoluciones del consumidor final.

Lista de pagos y conciliación (DEV - PROD)

Dos fuentes: API de lista de pagos y detalles del pedido / campos guardados por transacción. La lista informativa no sustituye el informe de conciliación, que es la fuente de verdad.

GET /api/cpgops-integrations/pos/v3/payments

Por defecto devuelve pagos de las últimas 24 horas, hasta 500 por solicitud. Las transacciones del mismo día pueden aparecer en D+1. Use start_date y end_date (AAAA-MM-DD) con limit (1–500) y skip para paginación.

Solicitud (curl)
curl --location 'https://microservices.dev.rappi.com/api/cpgops-integrations/pos/v3/payments?start_date=2024-02-16&end_date=2024-02-17&limit=500&skip=0' \
--header 'Authorization: {access_token}' \
--header 'Accept: application/json' \
--header 'x-rappi-channel: B2B'

Guarde para conciliación: order_id, amount, store_id, código de barras / código de pago, código de autorización, estado, created_at / updated_at (GMT 0).
