Differentiated discounts, loyalty banner, inventory mapping (Payless / Open Orders), user validation, and new user registration flows for Rappi partners.

Loyalty Integration Guide

What is a Loyalty Integration?

To provide a personalized and omnichannel experience to Rappi users who buy from a partner, we have loyalty programs that are an option to allow users to perceive all the benefits offered by the partner. This ensures that a user who is a frequent customer of a specific partner can find the benefits available in the Rappi app, make their product purchase decisions based on these benefits, and thus increase the value of orders generated and the frequency of purchase in those partners that have discounts from their loyalty programs available in Rappi.

Please note: currently this integration can only be implemented for partners that already have one of the following integrations: Connect, Payless, or Open Orders.

Checklist

  • Full understanding of the Customer Relationship Management (CRM) that, among other things, contains user data and available discounts.
  • Make the necessary adjustments to your systems to expose the mandatory information for validation.
  • Share with Rappi the mandatory fields to create new users from the Rappi app.
To keep in mind: loyalty refers to loyalty and membership programs that provide differentiated discounts. Loyalty discounts that Rappi handles at the moment are only simple discounts (e.g. 20% off). The new loyalty integration scheme allows automation of all process steps.

Integration components

  • Creation of differentiated discounts
  • User validation
  • New user registration

Loyalty banner

To generate the banner, the following are required:

  • A PNG image sized 176×132 px, up to 1 MB, for the program logo.
  • Banner title and banner body.
  • A web address (URL) of the partner loyalty program terms & conditions.

Loyalty discount mapping in the POS

Payless

In the Order Details endpoint, inside the customer object, there is a field called identification. This field shows the Loyalty ID for the user. The partner must read this ID to apply loyalty discounts correctly at checkout. If the value is null, no loyalty discounts should apply.

Open Orders

On the order creation event, the partner receives the user's fidelity_id in the event payload. If fidelity_id is empty (""), no loyalty discounts should apply.

Differentiated discounts in inventory

Loyalty discounts can be received within the conventional inventory integration (either API or SFTP, FULL or DELTA).

Important: an inventory integration must already be configured and working. The discount_price field carries non-differentiated discounts. Although Rappi has guardrails, these discounts should arrive only once at the beginning of the day.

If you use SFTP inventory integration

Send a new field loyalty_discount with the differentiated discount of the product (percentage). The field is only read if loyalty_discount is between 0 and 100 inclusive. This discount applies only to users enrolled in the loyalty program.

If you use API inventory integration

Send a new field loyalty_discount with the differentiated price of the product. The field is only read if it is less than the price field. This discount applies only to users enrolled in the loyalty program.

User Validation

The partner must validate the status of a given loyalty ID and its corresponding type in the loyalty program (if needed).

What do we need?

  • base_url: a webhook (HTTPS) where Rappi can validate the information.
  • headers: a key from the partner so only Rappi can access the partner's system (flat token).
Disclaimer: if the user validation request succeeds, Rappi immediately shows differentiated discounts when the partner has uploaded them correctly.

Validate endpoint

GET {base_url}/loyalty/membership/{user_id}

Required shape: the endpoint must be implemented exactly on this path structure so the integration works. It validates loyalty status using data provided in the Rappi app.

Considerations

  • A header with a flat token is required for authentication (you define the token format and header name).
  • The path must include the retailer identifier and the membership identifier pattern shown above.
  • user_id may be an ID number, loyalty code, email, or phone — whichever index your system needs.
  • Maximum wait time: 5 seconds. If exceeded, Rappi assumes a connection problem and cancels the request.
  • base_url must use HTTPS.
  • This flow does not issue access tokens; it uses only a flat token.
The response must be 200 OK with a body matching one of the examples below when membership is valid.
For good UX, average latency should be between 200 ms and 1 s.
Request (curl) 
curl --location --request GET '{base_url}/loyalty/membership/{user_id}' \
--header 'X-Api-Key: {{x-api-key}}'

The user_id used here is the ID stored for the user and is sent through other integrations such as Open Orders and Payless.

Response codes

Code Response Definition
200 OK Membership type (e.g. Gold, Diamond, Platinum) or the same user_id used to validate
404 NOT FOUND Loyalty ID not registered

Response body — 200 (membership types)

JSON 
{
  "type": "membership_type"
}

Response body — 200 (no levels)

JSON 
{}

New User Registration

For registration in a loyalty program, the partner must share the data required for onboarding so the user can complete registration directly in the Rappi app.

Disclaimer: if the new user inscription request succeeds, Rappi immediately shows differentiated discounts when the partner has uploaded them correctly.
ID vs phone: if the partner requires both ID and phone, the ID is the one shown in the Payless order details endpoint.

Inscription endpoint

POST {base_url}/loyalty/membership

Required shape: implement the endpoint exactly on this path structure.
Target average latency between 200 ms and 1 s for good UX.

Sends the required fields to enroll a new user in the loyalty program.

Considerations

  • Flat token in a header (you define format and header key).
  • No access-token generation; flat token only.
  • Specify which fields you need for the new user.
  • Maximum wait time: 5 seconds.
  • base_url must use HTTPS.
Request (curl) 
curl --location --request POST '{base_url}/loyalty/membership' \
--header 'X-Api-Key: {{x-api-key}}' \
--header 'Content-Type: application/json' \
--data '{
  "id": 1002485634,
  "name": "Pablo Lopez",
  "phone": "3185534870",
  "email": "pablol@gmail.com"
}'

The membership_id returned when onboarding succeeds is stored and sent through Open Orders and Payless.

Responses

Code Response Definition
201 CREATED Membership ID (string)
400 Bad Request Request body error
406 Not Acceptable User already exists
