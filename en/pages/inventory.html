Inventory

Update stock, prices, and simple discounts on Rappi via API or SFTP.

Getting started — via API

To update inventory through the HTTPS API, there are a few important points:

What is Inventory Integration?

Inventory integration exists to make it easier to update a retailer's stock and prices on Rappi — in this channel, through JSON calls over HTTP (datasets).

Requirements

  • Product catalog already created in Rappi.
  • Store list with store ID mapping between the retailer and Rappi.
  • API key (api_key) for the agreed environment (testing or production).

Authentication and format

Send api_key in the header. The body mainly contains the records array: each object represents one product in the store. The API may accept other fields (name, description, EAN, etc.), but only price, discount_price, and stock effectively update these values on Rappi's side according to your contract.

One store per request

Each call represents the inventory of a single store — all records must be aligned with this agreed scope for the project.

🚧 If more than one store is incorrectly sent in the same request, the endpoint may return an error.
👍 Request your test environment API key from your integration engineer.

Endpoints

POST /api/cpgs-integration/datasets

Batch inventory ingestion (datasets). Example base: https://services.grability.rappi.com/api/cpgs-integration/datasets.

Request (cURL) 
curl --location --request POST 'https://services.grability.rappi.com/api/cpgs-integration/datasets' \
--header 'Content-Type: application/json' \
--header 'api_key: {test_api_key}' \
--data-raw '{
    "type": "",
    "records": [
        {
            "id": "sku",
            "store_id": "store_id",
            "ean": "ean",
            "name": "productName",
            "description": "useful item",
            "trademark": "productTrademark",
            "price": 10.0,
            "discount_price": 5.0,
            "stock": 1,
            "sale_type": "U",
            "is_available": true,
            "image_url": "https://your-url.com/image"
        }
    ]
}'

Records (expected fields)

Reference table for the header and objects inside records[]. Keep names aligned with the contract agreed with your integration engineer.

Field Requirement Type Scope Definition
api_key Required string Header Unique integration token.
type Optional string Body Type as agreed (empty string or full/delta).
id Required string records[] Partner SKU / PLU.
store_id Required string records[] Store identifier (partner or Rappi).
ean Required string records[] Barcode (can be UPC).
name Required string records[] Product name.
trademark Required string records[] Trademark.
description Required string records[] Description (avoid unnecessary commas).
price Required number records[] Full price in local currency.
discount_price Optional number records[] Final discounted price.
stock Required integer records[] Units in stock.
sale_type Required string records[] Sale type.
is_available Required boolean / null records[] Expected in many contracts — see note below.
image_url Required string records[] Image URL.
👍 What to keep in mind:
  • is_available should be sent when agreed in the contract, but it typically does not change availability logic.
  • Optional fields can be empty or omitted, as agreed.
  • Fields outside the agreed list are not accepted.
  • The id field is prioritized together with EAN for internal matching.

Delta load

Send only items with relevant changes since the last sync. Others remain as they are. Include "type": "delta" in the root body as shown below.

Request (cURL · delta) 
curl --location --request POST 'https://services.grability.rappi.com/api/cpgs-integration/datasets' \
--header 'Content-Type: application/json' \
--header 'api_key: {test_api_key}' \
--data-raw '{
    "type": "delta",
    "records": [
        {
            "id": "sku",
            "store_id": "store_id",
            "ean": "ean",
            "name": "productName",
            "description": "useful item",
            "trademark": "productTrademark",
            "price": 10.0,
            "discount_price": 5.0,
            "stock": 1,
            "sale_type": "U",
            "is_available": true,
            "image_url": "https://your-url.com/image"
        }
    ]
}'
❗️ As with SFTP, it is not allowed to operate only with delta loads. Combine with at least one full load per day, preferably at the beginning of the business day.

👍 HTTP 201 Created: the body includes a job_id, status, and timestamp — other channels (email/internal notifications) follow what was agreed in the project.
Response example (JSON) 
{
    "job_id": "83360ac9-107a-4287-a081-5c1f12f81f66",
    "status": "created",
    "created_at": "2022-11-23T16:01:58.274215827Z"
}

💡 ❗ Missing required field — HTTP 400:

Response example (JSON) 
{
    "field": "store_id",
    "description": "the field is required",
    "internal": "required",
    "row": 0,
    "row_id": null
}

💡 ❗ Token or inactive data source — HTTP 400 (Bucket / data source inactive or non-existent):

Response example (JSON) 
{
    "message": "data_source does not exist or not active"
}

Getting started — via SFTP

What is Inventory Integration?

Inventory integration exists to make it easier to update a retailer's stock and prices on Rappi.

Integration requirements

  • Product catalog already created in Rappi.
  • Store list with store ID mapping between the retailer and Rappi.
  • PPK file (SSH key) for SFTP authentication.
  • Access to FileZilla or another compatible SFTP client.
  • Technical and operational capacity to integrate at least one full load per day.

The request (file upload)

In this mode, you must place the inventory file inside the designated folder on our SFTP server.

👍 Multiple stores in a single file: you can send several stores in the same file.

File name

The integration file name must follow one of the four formats below. If the upload is per store, the name must include the store ID for which stock is being reported. If the upload is a single file with all stores, use the partner name.

  • Date + time: STOREID_YYYY-MM-DD_HHmm.csv
  • Date only: STOREID_YYYY-MM-DD.csv
  • Date + time (all stores): PARTNER_NAME_YYYY-MM-DD_HHmm.csv
  • Date only (all stores): PARTNER_NAME_YYYY-MM-DD.csv

If you choose to send all stores in a single file, rows must be sorted by store.

Columns

Field Requirement Tipo Scope Definition
id Required string Body (row) Partner product ID (SKU, PLU).
store_id Required string Body (row) Partner or Rappi store identification code.
ean Required string Body (row) Barcode (can be UPC).
name Required string Body (row) Product name.
trademark Required string Body (row) Product trademark.
description Required string Body (row) Product description (avoid commas that break CSV).
price Required number Body (row) Full price in local currency.
discount_price Optional number Body (row) Price with discount applied (final price).
stock Required integer Body (row) Actual units in stock.
sale_type Required string Body (row) Sale type.
image_url Required string Body (row) Image URL.

The table above shows the fields that must be included in the integration. Below is a minimum, yet valid, record example for a successful integration — also follow file naming conventions.

CSV example (one row) 
id,store_id,ean,name,trademark,description,price,discount_price,stock,sale_type,image_url
SKU-DEMO-1,900408633,7891234567890,Sample product,Sample brand,Description without commas,10.99,9.49,120,U,https://cdn.example.com/img.jpg

CSV or TXT files may arrive without a header and without these column names. If the file does not include a standardized header, ask your integration engineer to map which column corresponds to which field.

👍 What to keep in mind
  • Discounts are fully optional, but they help attract users to stores in the Rappi app.
  • When not all products have discounts, you can repeat the full price in discount_price or leave it blank (avoid sending 0).
  • Non-required fields can be empty or can be omitted, depending on the agreement for your case.
  • Different fields or missing fields from the provided list are not accepted.
  • Although ean is required, our systems primarily use the id field to identify products in Rappi's database.

Integration types

There are two load types:

FULL

A full update must contain the entire catalog available in the store. Any product that does not appear in the file will be marked as unavailable, in addition to those already marked as unavailable in the file itself.

Full loads do not need a special file prefix, but you can also use the agreed integration prefix, for example full:.

DELTA

Delta updates must include only products whose stock (or relevant values) changed since the last update. Products not present in the file remain unchanged in the system.

Delta loads must indicate this type according to the agreed technical convention — usually through a delta prefix in the file name or submission; confirm the exact pattern with your integration engineer.

❗️ Partners cannot work only with delta loads. If they choose delta loads, they must send at least one full load per day. We recommend doing it at the beginning of the business day.

Testing SFTP

Your integration engineer will share the test-environment PPK with you so you can test the connection. During tests, always use the same products and the same store agreed by both parties.

📘 Template: your integration engineer provides a download link (for example "Download here") with the sample file used in tests.

After uploading, inform the integration engineer so they can verify in the pipeline whether processing completed correctly.

❗ Important: unlike the API, you will not receive a response in FileZilla just by completing the upload — the integration result follows the asynchronous flow notified by Rappi.

Important notes

First API update (production)

Before go-live, confirm requirements, SFTP/API tests, and payloads with your engineer. In production, use the API_KEY released only after formal authorization.

Combined FULL + delta usage

Both in API and SFTP, avoid working only with delta loads: combine them with at least one daily full load, preferably in the morning.
